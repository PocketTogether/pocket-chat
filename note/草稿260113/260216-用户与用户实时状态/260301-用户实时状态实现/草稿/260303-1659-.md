#### useSelfPresenceStore

```
文件：src/stores/self-presence.ts
名称：useSelfPresenceStore（setup 语法的 pinia store）

state 中的变量（均为 ref 或 computed）：
- isViewing: Ref<boolean>
  表示当前标签页是否“认为是可见状态”。初始值可根据 document.visibilityState === 'visible' 来设定。
  在 visibilitychange 事件中更新：visible -> true，hidden -> false。
  供循环判断与发送逻辑、以及其他组件（如 UI 显示“当前是否在前台”）使用。

- isTyping: Ref<boolean>
  表示当前用户是否正在输入。由 useSelfPresenceTypingReporter 更新。
  循环中会读取此值，决定是否需要发送 isTyping: true 的状态。

- lastSentPayload: Ref<{ isNotViewing: boolean; isTyping: boolean } | null>
  记录上一次成功发送到 userPresencesStatus 集合的“精简 payload”。
  用于在下一次判断时，与当前“待发送 payload”做浅比较，如果完全相同则不再发送。

- lastSentAt: Ref<number | null>
  记录上一次发送 userPresencesStatus 成功完成时的时间戳（Date.now()）。
  用于节流：如果当前时间 - lastSentAt < 2000ms，则本轮判断不发送。

- isSending: Ref<boolean>
  表示当前是否有一次 userPresencesStatus 的发送正在进行中。
  在发送前设为 true，发送完成（成功或失败）后设为 false。
  循环与其他调用方在判断时，如果 isSending 为 true，可以选择直接跳过本轮发送。

- loopStarted: Ref<boolean>
  表示无限循环是否已经启动，避免在 app.vue 或其他地方重复启动。
  useSelfPresenceLoop 在启动时会先检查此值。

- lastNotViewingMarkSentAt: Ref<number | null>
  记录最近一次通过 sendBeacon 向 userNotViewingMarks 集合发送的时间戳。
  visibilitychange(hidden) 时会参考此值，避免在短时间内重复发送“离开标记”。

- selfUserId: Ref<string | null>
  当前登录用户的 id，用于构造发送 payload 的 author 字段。
  一般在 auth 相关逻辑中初始化或更新。

getters / computed：
- currentPayload: Computed<{ isNotViewing: boolean; isTyping: boolean }>
  根据 isViewing 和 isTyping 计算出当前应该发送给 userPresencesStatus 的 payload：
  isNotViewing = !isViewing
  isTyping = isTyping
  这样外部只需要读取 currentPayload 即可得到“当前语义上的状态”。

actions：
- setViewing(next: boolean): void
  更新 isViewing。一般由 useSelfPresenceLoop 中的 visibilitychange 监听调用。
  也可以在某些特殊场景下由组件主动调用（例如强制认为当前不可见）。

- setTyping(next: boolean): void
  更新 isTyping。由 useSelfPresenceTypingReporter 调用。
  输入栏组件只需要调用 typingReporter 的方法，不直接操作 store。

- markLoopStarted(): void
  将 loopStarted 设为 true。由 useSelfPresenceLoop 在启动时调用。

- updateLastSent(payload: { isNotViewing: boolean; isTyping: boolean }): void
  在 userPresencesStatus 发送成功后调用，更新 lastSentPayload 和 lastSentAt。
  lastSentAt = Date.now()。

- setSending(next: boolean): void
  在发送 userPresencesStatus 前后调用，标记 isSending。

- updateLastNotViewingMarkSent(): void
  在成功通过 sendBeacon 发送 userNotViewingMarks 后调用，更新 lastNotViewingMarkSentAt = Date.now()。

使用方式：
- 组合式函数 useSelfPresenceDispatcher / useSelfPresenceLoop / useSelfPresenceTypingReporter 中通过 useSelfPresenceStore() 获取同一个实例。
- 组件层（如输入栏、app.vue）不直接操作复杂逻辑，只调用 store 的简单 action 或组合式函数暴露的方法。
```

#### useSelfPresenceDispatcher

```
文件：src/composables/self-presence/dispatcher.ts
名称：useSelfPresenceDispatcher

依赖：
- useSelfPresenceStore
- pbUsersPresencesStatusCreateApi
- pbUsersNotViewingMarksCreateApiBySendBeacon

内部变量（通过 store 读取，不单独再定义 ref）：
- store.isViewing
- store.isTyping
- store.currentPayload
- store.lastSentPayload
- store.lastSentAt
- store.isSending
- store.lastNotViewingMarkSentAt
- store.selfUserId

内部函数：
- buildPresencePayload(): { author: string; isNotViewing: boolean; isTyping: boolean }
  从 store.currentPayload 和 store.selfUserId 构造发送给 pbUsersPresencesStatusCreateApi 的 payload。
  只负责组装数据，不做节流与比较逻辑。

- shouldSkipPresenceSend(now: number, nextPayload: { isNotViewing: boolean; isTyping: boolean }): boolean
  判断本次是否应该跳过发送 userPresencesStatus：
  1. 如果 store.selfUserId 为空，直接跳过。
  2. 如果 store.isSending 为 true，直接跳过（避免并发）。
  3. 如果 store.lastSentAt 存在且 now - lastSentAt < 2000ms，跳过（节流）。
  4. 如果 store.lastSentPayload 存在且与 nextPayload 浅比较完全相同，跳过（状态未变化）。
  返回 true 表示跳过，false 表示可以发送。

- sendPresenceOnce(): Promise<void>
  核心发送函数，供循环与其他地方调用。
  步骤：
  1. const now = Date.now()
  2. const nextPayload = { isNotViewing: !store.isViewing, isTyping: store.isTyping }
  3. 如果 shouldSkipPresenceSend(now, nextPayload) 为 true，则直接 return。
  4. 调用 store.setSending(true)。
  5. 调用 pbUsersPresencesStatusCreateApi({ author: store.selfUserId!, isNotViewing: nextPayload.isNotViewing, isTyping: nextPayload.isTyping })。
  6. 成功后调用 store.updateLastSent(nextPayload)。
  7. finally 中调用 store.setSending(false)。

- sendNotViewingMarkIfNeeded(): void
  用于在 visibilitychange(hidden) 时调用，内部使用 pbUsersNotViewingMarksCreateApiBySendBeacon。
  步骤：
  1. const now = Date.now()
  2. 如果 store.selfUserId 为空，直接 return。
  3. 根据 store.lastSentAt 和 store.lastSentPayload 判断当前是否已经“会被认为是 isNotViewing: true”：
     - 如果 lastSentPayload?.isNotViewing === true 且 now - lastSentAt < 100000（或一个合理的阈值），
       则认为即使不再发送 userNotViewingMarks，也会被判定为不显示，可直接 return。
  4. 再根据 store.lastNotViewingMarkSentAt 做一个简单节流，例如 now - lastNotViewingMarkSentAt < 2000ms 时跳过。
  5. 构造 body：{ author: store.selfUserId! }。
  6. 调用 pbUsersNotViewingMarksCreateApiBySendBeacon(body)。
  7. 如果 sendBeacon 返回 true（或认为已成功），调用 store.updateLastNotViewingMarkSent()。

- triggerPresenceCheckNow(): Promise<void>
  对外暴露的“立即判断并尝试发送一次”方法。
  内部只是简单调用 sendPresenceOnce()。
  供：
  - 无限循环每 5 秒调用；
  - useSelfPresenceTypingReporter 在“第一次输入”或“输入状态从 false -> true”时调用；
  - 其他需要立即刷新状态的地方调用。

对外暴露：
- return { triggerPresenceCheckNow, sendNotViewingMarkIfNeeded }

使用方式：
- useSelfPresenceLoop 中在循环体内调用 triggerPresenceCheckNow()。
- useSelfPresenceLoop 中在 visibilitychange(hidden) 事件中调用 sendNotViewingMarkIfNeeded()。
- useSelfPresenceTypingReporter 在特定时机调用 triggerPresenceCheckNow()。
```

#### useSelfPresenceLoop

```
文件：src/composables/self-presence/loop.ts
名称：useSelfPresenceLoop

依赖：
- useSelfPresenceStore
- useSelfPresenceDispatcher

内部变量：
- store = useSelfPresenceStore()
- dispatcher = useSelfPresenceDispatcher()
- loopTimerId: number | null
  用于记录 setTimeout / setInterval 的 id，方便在必要时清理。
  这里建议使用递归 setTimeout 而不是 setInterval，以便更精细控制。

内部函数：
- startLoopIfNeeded(): void
  启动无限循环与 visibilitychange 监听，只在首次调用时生效。
  步骤：
  1. 如果 store.loopStarted 为 true，则直接 return。
  2. 调用 store.markLoopStarted()。
  3. 调用 startVisibilityListener()。
  4. 调用 scheduleNextTick() 启动递归循环。

- scheduleNextTick(): void
  使用 window.setTimeout(() => { loopTick() }, 5000) 安排下一次循环。
  将返回的 id 存入 loopTimerId。

- loopTick(): Promise<void>
  无限循环的单次执行逻辑：
  1. 调用 dispatcher.triggerPresenceCheckNow()。
  2. 在 finally 中再次调用 scheduleNextTick()，保证循环继续。

- startVisibilityListener(): void
  注册 window.addEventListener('visibilitychange', onVisibilityChange)。
  onVisibilityChange 中：
  1. const isVisible = document.visibilityState === 'visible'
  2. 调用 store.setViewing(isVisible)
  3. 如果 isVisible 为 false：
     - 调用 dispatcher.sendNotViewingMarkIfNeeded()
     - 也可以选择调用 dispatcher.triggerPresenceCheckNow()，但要注意节流逻辑已经在 dispatcher 内部处理。
  4. 如果 isVisible 为 true：
     - 可以选择调用 dispatcher.triggerPresenceCheckNow()，让“重新可见”尽快同步到后端。

- stopLoop(): void（可选）
  如果需要在某些场景下停止循环，可以提供此函数：
  1. 如果 loopTimerId 不为 null，调用 clearTimeout(loopTimerId)。
  2. 移除 visibilitychange 监听。
  3. 将 store.loopStarted 设为 false（可选，看是否允许重新启动）。

对外暴露：
- return { startLoopIfNeeded, stopLoop }

使用方式：
- 在 app.vue 的 setup 中调用：
  const { startLoopIfNeeded } = useSelfPresenceLoop()
  onMounted(() => {
    startLoopIfNeeded()
  })
- 一般只在全局入口调用一次，确保当前用户的状态循环在整个会话期间持续运行。
```

#### useSelfPresenceTypingReporter

```
文件：src/composables/self-presence/typing-reporter.ts
名称：useSelfPresenceTypingReporter

依赖：
- useSelfPresenceStore
- useSelfPresenceDispatcher

内部变量：
- store = useSelfPresenceStore()
- dispatcher = useSelfPresenceDispatcher()
- lastTypingValue: Ref<boolean>
  用于记录上一次对外暴露的“输入状态”，用于判断是否从 false -> true。

内部函数：
- reportTyping(isTyping: boolean): void
  供输入栏组件在“输入状态变化”时调用。
  步骤：
  1. 如果 store.isTyping === isTyping，则直接 return（避免重复写入）。
  2. 调用 store.setTyping(isTyping)。
  3. 如果 isTyping 为 true 且 lastTypingValue.value === false：
     - 说明是“从非输入状态进入输入状态”的第一次输入。
     - 调用 dispatcher.triggerPresenceCheckNow()，让“正在输入”尽快同步到后端。
  4. 更新 lastTypingValue.value = isTyping。

- notifySentMessage(): void
  供输入栏在“发送消息成功后”调用。
  语义：用户刚刚完成一次输入并发送，可以视为一次“活跃操作”，可以考虑立即刷新状态。
  步骤：
  1. 可选：将 store.setTyping(false)，视具体产品逻辑而定。
  2. 调用 dispatcher.triggerPresenceCheckNow()，让“刚刚活跃过”的状态尽快同步。

- attachToInputElement(el: HTMLTextAreaElement | HTMLInputElement): void（可选）
  如果希望在组合式函数内部直接绑定 DOM 事件，可以提供此方法：
  1. 在 el 的 input 事件中调用 reportTyping(true)。
  2. 在 blur 事件中调用 reportTyping(false)。
  这样组件层只需要在模板中通过 v-bind:ref 或指令把元素传进来。

对外暴露：
- return { reportTyping, notifySentMessage, attachToInputElement }

使用方式：
- 在输入栏组件中：
  const { reportTyping, notifySentMessage } = useSelfPresenceTypingReporter()
  在 input 事件中调用 reportTyping(true)
  在 blur 或输入清空时调用 reportTyping(false)
  在消息发送成功后调用 notifySentMessage()
```

### 其他细节的描述

```
1. 关于时间阈值
- 2000ms 的“发送节流”与“sendBeacon 节流”可以抽成常量，例如：
  const PRESENCE_MIN_INTERVAL = 2000
  const NOT_VIEWING_MARK_MIN_INTERVAL = 2000
- “100 秒内无记录视为离线”的逻辑主要在后端或“全部用户的实时状态订阅接收与处理”那一块，
  这里只需要保证前端在正常活跃时能稳定每 50 秒以内发送一次即可。

2. 关于状态语义
- store 中只维护 isViewing / isTyping 两个“语义清晰”的布尔值。
- 发送给 userPresencesStatus 时统一转换为 isNotViewing = !isViewing。
- userNotViewingMarks 只需要 author，不需要 isTyping 等信息。

3. 关于可扩展性
- 如果未来要加入“当前播放列表”等字段，可以在 store 中增加 currentPlaylistId 等字段，
  在 buildPresencePayload 中一并加入，且在 lastSentPayload 中也记录，保持“比较逻辑”一致。
- useSelfPresenceTypingReporter 也可以扩展为“输入内容长度”等更细粒度的状态，但发送逻辑仍由 dispatcher 统一处理。

4. 关于错误处理
- pbUsersPresencesStatusCreateApi 失败时，sendPresenceOnce 只是在 finally 中重置 isSending，
  不更新 lastSentPayload / lastSentAt，这样下一轮循环仍会尝试发送。
- pbUsersNotViewingMarksCreateApiBySendBeacon 一般没有回调结果，
  可以根据 sendBeacon 的返回值（boolean）简单判断是否认为“已发送”，
  若返回 false，可以选择不更新 lastNotViewingMarkSentAt，让下一次有机会重试。

5. 关于与“全部用户的实时状态订阅接收与处理”的配合
- 当前模块只负责“自己”的状态发送，不关心订阅。
- 订阅模块只需要根据 userPresencesStatus / userNotViewingMarks 的记录时间与内容，
  按你之前设想的规则合并出“最终状态”，即可。
```
