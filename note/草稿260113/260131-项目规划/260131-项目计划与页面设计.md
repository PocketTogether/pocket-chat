## pb数据库设计

messages 消息集合的一些修改
```
mentionedUsers 新增关系字段，关系至users集合，是可多的，用于记录本消息所@的用户
mentionMap 新增json字段，用于记录消息中用户名所对应的用户id，键值对，需要这个主要是用来明确指定消息内容中 @xxxx（username）所指定的用户，因为username是可能被用户修改的
```

usersPresencesStatus 用户实时状态集合
```
字段
isNotViewing 是否不在看（是否屏幕正在不显示）、布尔
isTyping 是否正在输入、布尔
author 用户关系字段

用户状态集合
用于实现用户状态：
- 在线状态、在线但屏幕未显示状态、输入状态、离线状态
还附带用户当前的一些信息
- 用户当前正在播放的播放列表（关系字段）（目前这一阶段的开发暂不需要，以后再说）

前端至少每50秒会创建一条这个记录
当某一用户最近一条记录都是100秒以前的，就会判断其为不在线
更细节的逻辑
前端有一个间隔5秒的死循环，其中判断状态（如是否正在输入这种的，是基本在store里，ui中会操作store）
前端会记录上次发送的具体数据
前端会记录上次发送的完成时的时间戳
前端会记录正在发送的具体数据，为null时即为非正在发送
死循环判断状态，如果状态变化就发送
如果 上次发送的完成时的时间戳 距今小于两秒，本次就什么也不干
建议这种判断和发送逻辑封装一下，因为不仅仅可能在死循环中调用，还可能供其他vue文件或组合式函数中调用
其可能供其他vue文件或组合式函数中调用，比如输入栏用户开始输入第一个字时立即调用一下

```

usersNotViewingMarks 用户未查看标记集合
```
字段
author 用户关系字段

是在 visibilitychange hidden 时会向此集合创建记录
因为用的是 navigator.sendBeacon 的post不能有header所以没办法带用户鉴权token
所以此集合api规则不强制指定用户关系字段必为创建者，即都是不带鉴权来创建的
你可能会有疑惑 用户状态集合 已经有 “字段 是否屏幕正在显示、布尔” 这样的疑惑
这是因为此用户离开事件集合是专门用于 navigator.sendBeacon，
即，其实主要目的是，用户关闭标签页前能迅速在此创建一条记录，visibilitychange 没办法准确判断是 用户关闭标签页，所以用户切换标签页或最小化窗口等操作时也会，为此我想了即使这样也没关系的办法
用户离开事件集合，就让它也确实代表是否屏幕正在显示，即有用户在此创建记录，即代表其已不显示，这样当用户关闭标签页时，能达到其马上状态就是不显示的效果，毕竟没办法准确判断其关闭标签页，权宜之计

如果同用户所指的记录中：
- 用户离开事件集合的个记录的创建时间更大，那就指的是用户是屏幕正在不显示状态
- 用户状态集合的个记录的创建时间更大，那就以其内容为准

一些逻辑细节
前端会记录上次发送的完成时的时间戳
在 navigator.sendBeacon 之前，还会根据当前这一块“前端会记录上次发送的完成时的时间戳” 还有用户状态这一块 “前端会记录上次发送的具体数据 前端会记录上次发送的完成时的时间戳” 来判断到底是否有必要发送，即如果已会被判断为已不显示就没必要再发送
```
