# 🧠 用户状态集合的描述（重写、结构化、可直接放文档）

```
用户状态集合（userPresences）用于记录用户的实时状态快照。

前端每隔约 50 秒会创建一条新的状态记录，用于维持用户在线状态。
此外，前端在状态变化（如开始输入、停止输入、屏幕显示状态变化）时也会立即创建记录。

字段说明：
- user：关联到 users 集合，表示该状态属于哪个用户。
- isScreenVisible：布尔值，表示当前标签页是否处于可见状态。
- isTyping：布尔值，表示用户是否正在输入。

在线判断逻辑：
- 如果某用户最近一条 userPresences 记录的创建时间在 100 秒以内，则视为在线。
- 超过 100 秒未更新，则视为离线。

前端逻辑摘要：
- 前端有一个 5 秒的循环，用于检测状态变化。
- 前端会记录上次发送的状态数据与发送完成时间戳。
- 若状态未变化，或距离上次发送完成时间不足 2 秒，则不会发送。
- 状态变化时（例如用户开始输入第一个字），会立即发送一次。
- 该逻辑会被封装成可复用函数，以便在组件或组合式函数中调用。
```

# 🧠 用户离开事件集合的描述（重写、结构化、可直接放文档）

```
用户离开事件集合（userVisibilityEvents）用于记录用户在标签页变为 hidden 时的事件。

由于 navigator.sendBeacon 无法携带鉴权头，因此该集合的 API 规则允许匿名创建记录，不强制要求 user 字段必须等于创建者。

该集合的主要用途：
- 在用户关闭标签页前，通过 visibilitychange + sendBeacon 快速创建一条记录。
- 因为无法准确判断用户是否关闭标签页（visibilitychange 在切换标签页、最小化等情况也会触发），因此采用“只要创建记录就视为屏幕不显示”的策略。

状态判定逻辑：
- 若某用户在 userVisibilityEvents 中最新记录的时间 > userPresences 中最新记录的时间，则认为用户当前屏幕不显示。
- 否则，以 userPresences 的 isScreenVisible 字段为准。

前端逻辑摘要：
- 前端会记录上次状态发送完成时间戳。
- 在触发 sendBeacon 前，会根据当前状态判断是否有必要发送（例如已经被判定为不显示时就不再发送）。
```

# ✅ 三、两个集合之间的关系总结（你设计得非常好）

|情况|判定方式|
|---|---|
|用户是否在线|看 userPresences 最新记录是否在 100 秒内|
|屏幕是否显示|比较 userVisibilityEvents 与 userPresences 的最新记录时间|
|是否正在输入|由 userPresences 最新记录的 isTyping 决定|

你构建的是一种 **“事件 + 状态快照” 双轨模型**：

- **userPresences**：主状态源
- **userVisibilityEvents**：补充事件源（解决 sendBeacon 无法鉴权的问题）

这套模型非常健壮，也非常适合 PocketBase。
