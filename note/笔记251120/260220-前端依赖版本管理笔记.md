# 📘 前端依赖版本管理笔记

（Vue3 / Vite / pnpm / package.json）

前端项目里最容易被忽视、但对工程稳定性影响巨大的，就是 **依赖版本号策略**。  
尤其是使用 pnpm 时，理解版本号前缀（^、~、无符号）的机制，对构建可复现性至关重要。

这篇笔记分为三部分：

1. `"sass": "1.87.0" / "^1.87.0" / "~1.87.0"` 的区别
2. 如何让 `pnpm add` 不再自动加 `^` 或 `~`
3. 是否有很多团队这样做

---

# 1. 版本号写法的含义

语义化版本（semver）格式：

```
MAJOR.MINOR.PATCH
主版本.次版本.补丁版本
```

以 `1.87.0` 为例。

## 无符号：`"sass": "1.87.0"`

**完全锁死版本**  
只能安装 1.87.0，不会自动升级。

适合追求：

- 构建可复现
- 零隐式升级
- CI 稳定性

## 波浪号：`"sass": "~1.87.0"`

**允许升级 patch，不允许升级 minor**

范围：

```
>=1.87.0 <1.88.0
```

适合：

- 想要 bug 修复
- 又不想冒 minor 风险

## 插入符号：`"sass": "^1.87.0"`

**允许升级 minor + patch，不允许升级 major**

范围：

```
>=1.87.0 <2.0.0
```

适合：

- 默认行为
- 想自动获得新特性和修复

# 2. 如何让 pnpm add 不再自动加 ^ 或 ~

pnpm 默认会在 package.json 写入 `^`，但你可以关闭它。

## 项目级设置（推荐）

在项目根目录创建 `.npmrc`：

```
save-exact=true
```

效果：

```
pnpm add vue
```

写入：

```
"vue": "3.5.16"
```

不会再出现 `^` 或 `~`。

## 全局设置（所有项目生效）

```
pnpm config set save-exact true -g
```

## 单次使用

```
pnpm add vue --save-exact
```

# 3. 很多人这样做吗？

答案分两类。

## ✔ 严肃工程团队（大量）

这些团队通常会 **禁止 ^ 和 ~**：

- 金融行业
- 医疗行业
- 大型 SaaS
- 需要审计、合规、可复现构建的公司
- 追求 deterministic build 的工程师

他们的策略通常是：

- package.json 使用精确版本
- pnpm-lock.yaml 锁死依赖树
- CI 禁止隐式升级
- 所有依赖升级必须通过 PR 审核

你现在采用的方式正是这类团队的标准做法。

## ✔ 普通开源项目 / 小团队

他们通常保留 `^`，因为：

- 想自动获得 bug 修复
- 不太关心可复现性
- 不想频繁手动升级依赖

但这类项目经常出现“依赖漂移导致构建失败”的问题。

---

# 📌 总结

- `"1.87.0"`：完全锁死版本，最稳定
- `"~1.87.0"`：只升级 patch
- `"^1.87.0"`：升级 minor + patch
- 想让 pnpm add 不加 ^ 或 ~ → `.npmrc` 写 `save-exact=true`
- 严肃工程团队大量采用精确版本策略

你现在的做法（精确版本 + pnpm）是 **最可控、最工程化、最可复现** 的依赖管理方式。
