# 📝 **笔记：为什么数据处理应该放在 queryFn，而不是组合式函数中？**

在使用 Vue + Vue Query（或 TanStack Query）构建复杂业务时，我们经常会遇到一个问题：

> **数据清洗、聚合、转换逻辑应该放在组合式函数（composable）里，还是放在 queryFn 里？**

一开始我把这些逻辑放在组合式函数中，例如根据用户 ID 构建最新状态 Map、最早状态 Map 等。但随着组件数量增多，我发现这种方式有一些明显的问题。

这篇笔记记录了我对这个问题的思考过程，以及最终为什么选择把数据处理放到 queryFn 中。

---

## 🧩 **方案 A：数据处理放在组合式函数中（原方案）**

### **优点**

- 逻辑集中在一个 composable 中，易于维护
- 不污染 queryFn，使其保持“纯数据获取”

### **缺点**

- **每次调用 composable 都会重新执行数据处理**
- 如果子组件也调用 composable，会导致 **重复计算**
- 父组件必须传递处理后的数据，子组件无法独立使用 composable
- 不够“组合式”，因为组合式函数变成了“依赖父组件的逻辑容器”

换句话说：

> **组合式函数变成了“视图层的加工厂”，而不是“可复用的逻辑单元”。**

这违背了组合式的核心理念：  
**每个组件都应该能独立调用 composable，而不是依赖父组件传值。**

---

## 🧩 **方案 B：数据处理放在 queryFn 中（最终选择）**

### **为什么更好？**

### 1. **天然共享缓存，不会重复计算**

Vue Query 的 queryFn：

- 只在缓存失效时执行
- 多个组件调用同一个 useQuery → 共享同一份缓存
- 数据处理也会被缓存

这意味着：

> **数据清洗只做一次，全局共享。**

这正是我想要的效果。

---

### 2. **子组件可以独立调用 composable，不依赖父组件**

如果数据处理放在 queryFn：

- 子组件只需要传入 userId
- composable 内部自动从缓存中获取处理后的数据
- 不需要父组件传递任何加工后的数据

这才是真正的组合式：

> **组件之间不需要互相依赖，只依赖 composable。**

---

### 3. **职责划分更清晰：数据层 vs 视图层**

现代前端架构（React Query、SWR、Apollo、Relay）都强调：

- **queryFn = 数据层（负责数据获取 + 清洗）**
- **composable = 视图逻辑层（负责组合 store + 提供计算属性）**

把数据处理放在 queryFn 中：

- 数据层负责“把数据变成可用的形状”
- 视图层只负责“如何使用这些数据”

这让整个系统更清晰、更可维护。

---

### 4. **工程上更方便，也是更多人的选择**

在大型项目中，大家普遍会：

- 在 queryFn 中完成数据清洗
- 在 composable 中组合多个 query/store
- 在组件中直接调用 composable

因为这样：

- 性能更好（缓存）
- 逻辑更清晰（职责分离）
- 组件更独立（组合式理念）
- 更符合现代数据层的最佳实践

---

# 🎯 **最终结论**

> **把数据处理放在 queryFn 中，是更符合组合式理念、更工程化、更高性能的做法。**

理由总结如下：

|维度|组合式函数处理|queryFn 处理|
|---|---|---|
|缓存|❌ 无缓存，重复计算|✔️ 天然缓存，共享结果|
|子组件独立性|❌ 依赖父组件传值|✔️ 子组件可独立调用|
|组合式理念|❌ 组合式函数变得臃肿|✔️ composable 更纯粹|
|工程实践|❌ 容易重复逻辑|✔️ 更符合现代数据层设计|
|性能|❌ 多次执行|✔️ 只执行一次|

因此：

### ✔️ 数据清洗、聚合、转换 → 放在 queryFn

### ✔️ composable → 只负责组合 store/query + 提供计算属性

### ✔️ 子组件 → 只依赖 composable，不依赖父组件

这才是最“组合式”、最工程化、最现代的架构方式。

## ✔️ 这是工程上非常干净的解决方案

下面是详细原因。

---

# 1. 为什么 thisTimeQueryUuid 是“必要”的？

我有两个查询：

### ① 原始查询

获取 500 条用户状态：

```ts
useUsersPresencesStatusInitGetListQuery()
```

### ② 派生查询（Derived Query）

对原始数据进行加工：

- latestMap
- earliestMap

```ts
useUsersPresencesStatusInitGetListWithMapByUserProcessQuery()
```

派生查询依赖原始查询的数据。

问题来了：

## ❗派生查询的 queryKey 必须随着原始数据变化而变化

否则：

- 原始数据更新了
- 派生查询的缓存却不会失效
- 组件会继续使用旧的 Map
- 逻辑错误且难以排查

因此：

> **派生查询必须知道“原始查询是否发生了变化”。**

而原始查询返回的数据结构中，并没有一个天然的“版本号”或“变化标识”。

于是：

## ✔️ thisTimeQueryUuid 就成为了“变化标识（version tag）”

每次原始查询成功：

- 生成一个新的 uuid
- 派生查询的 queryKey 也随之变化
- 派生查询自动重新计算
- 缓存完全正确

这使得派生查询的缓存行为完全可靠。

---

# 2. 为什么 thisTimeQueryUuid 是“合理”的？

因为它满足了派生查询的两个核心需求：

### ① 唯一性（uniqueness）

每次查询成功 → uuid 不同 → 派生查询缓存自动失效。

### ② 稳定性（stability）

只要原始查询的数据不变 → uuid 不变 → 派生查询不会重复执行。

这正是 Vue Query 的缓存机制所需要的。

---

# 3. 为什么 thisTimeQueryUuid 是“巧妙”的？

因为它解决了一个常见但隐蔽的问题：

> **如何让派生查询自动跟随原始查询的变化？**

常见但不够优雅的方式包括：

- 用 JSON.stringify(data) 作为 queryKey（性能差）
- 用 data.length 或 data[0].id 作为 key（不可靠）
- 用时间戳（不稳定）
- 手动触发 invalidateQueries（侵入性强）

而 thisTimeQueryUuid：

- 完全自动
- 完全稳定
- 完全无侵入
- 完全不依赖数据结构
- 完全不影响原始查询的缓存
- 完全符合 Vue Query 的哲学

可以说是：

## ✔️ “最小代价换取最大正确性”的设计

---

# 4. 这种方式在社区中常见吗？

是的。

在 React Query / Vue Query 社区中，很多人使用类似的方式：

- version
- revision
- generation
- hash
- uuid

这些字段的作用都是：

> **让派生查询的缓存与原始查询的生命周期保持一致。**

你的做法属于其中最干净、最稳定的一类。

## ✔️ thisTimeQueryUuid 是必要的
